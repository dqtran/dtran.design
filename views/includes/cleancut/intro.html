WHAT

While I was a Creative Technology Apprentice at Barbershop, my project was to create Clencut, an internal Q&A tool. Barbershop is a creative technology studio based out of Denver, Colorado that focus on building brands and accelerating startups through digital products.

I was responsible for.

Product design. User Experience and User Interface Design. Front-end development.

{% include "includes/assets/empathize.html" %}

I started the project by understanding the requirements from the Barbershop team for the project. Based on the app development process the team and the scope of the project, I started by doing a short research about other tools in the market and interviewing few people about their current Q&A process at their firm.

INSIGHTS

The current Q&A process is time-consuming and repetitive.

The Q&A teams are using task management tools such as Trello to create a test plan, issues then share them to the dev team.

Testing on the multi platforms, devices and viewports application is time-consuming and expensive.

{% include "includes/assets/define.html" %}

After gathering all the requirements and discovering key insights, I narrowed down to the core requirements and features for the project, and defined project goals.

REQUIREMENTS

The application must integrate with Github or Bitbucket.

The application allow the users to create test plans, tasks and create Github or Bitbucket issues with custom tags.

The test plans and tasks can be accessible within the application interface based the view port and different web browser.

{% include "includes/assets/ideate.html" %}

I started the project by creating many sketches about the app to generate ideas about the user flow of the application.

First, I designed the "creating test plan and task" flow. The user should able to login using their Github or Bitbucket account, select the application that they want to work with, then start creating the first test plan and tasks associate with the application. The user should also able to add staging or production URL and tags to used for each tasks under Settings. And the user can also see the current progress of the test plan and which browsers are working or failing.

Second, I started playing around with the user interface where users can access test plans and tasks within the application user interface. I explored the user inteface of Intercom and ProductHunt and was doing some sketches.

Given a limited hours that I had per week for the project, I skipped the wireframes and jump right into creating high fidelity mockups in order to meet the project milestones. I started to explore some of the desktop user interface practices from Apple and Invision. Then I played with the color palette and typography. Red, black and white was the chosen color pallete and "Source Sans Pro" was the primary font.

Based on the feedbacks and insights from the team, I went back to re-define the solution with new requirements.

REQUIREMENTS

The application won't be used by QA person only. It will be a platform for developer, QA person and the client to collaborate and test the application.

The user should able to create multiple release plans and copy test plans from previous release.

The user should able to add details about the tasks and any attachements or design assets.

{% include "includes/assets/ideate.html" %}

I began to draw a user flow of the application based on the new requirements to get better understanding of how I a going to organize the contents. For this redesign process, I skipped sketches and jump right into creating high fidelity mockups.

First, I redesigned the user flow of the application. The user should still able to login using their Github or Bitbucket account, select the application that they want to work with, then start creating the first release version of the application. The user then can start creating new test plan, tasks and details information about the task. Once the user create a new release plan,they will have an option to add test plans or tasks from the previous release.

Second, I started playing around with the red "add button". I really like the concept of the floating button from the Google Material Design.The add button is now located at the same location of the application. The size of the button will also help indicating whether you are adding a release plan, a test plan or a task.

Finally, I create two different views: task view and QA view. The developer will use the task view to create test plans and tasks. Once they are done with a specific task or test plan, they can mark it as "complete" and the task will be available under QA view. The task is now ready to be tested.

LESSON LEARNED

This project taught me how to carefully think about the target audience for the application. Muliple users with different goals in mind will use your application so I make sure I design the application that is easy to use by all of our target audience.

I also learned that even though there is a limited hours that I have per week for the project, I should have done some wireframes. However, since I only work with a small Barbershop team, it is efficient enough to collabrorate and share my ideas without wireframes. It also proves that the design thinking process is very flexible and can repeate as much as we can. It is a very agile process.
