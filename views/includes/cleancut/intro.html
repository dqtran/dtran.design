WHAT

Barbershop is a creative technology studio based out of Denver, Colorado that focus on building brands and accelerating startups through digital products. I was a Creative Technology Apprentice at Barbershop.My project was to create Clencut, an internal Q&A tool.

I was responsible for.

Product design. User Experience and User Interface Design. Front-end development.

{% include "includes/assets/empathize.html" %}

I started by looking at the project requirements from the Barbershop team. I then did some research about other tools in the market and interviewed few people about their current Q&A process at their firm. This research was based on the app development process of the team and the scope of the project.

INSIGHTS

  • The current Q&A process is time-consuming and repetitive.
  • The Q&A teams are using task management tools such as Trello to create a test plan, issues then share them to the dev team.
  • Testing on the multi platforms, devices and viewports application is time-consuming and expensive.
{% include "includes/assets/define.html" %}

After the discovery of these key insights, I narrowed the core requirements and features for the project, and defined the project's goals.

REQUIREMENTS

  • The application must integrate with Github or Bitbucket.
  • The application allow the users to create test plans, tasks and create Github or Bitbucket issues with custom tags.
  • The test plans and tasks can be accessible within the application interface based on the view port and different web browsers.
{% include "includes/assets/ideate.html" %}

I started the ideation phase by creating many sketches. This helped to generate ideas about the user flow of the application.

First, I designed the â€œCreate Test Plan and Task" flow. The user should be able to login using their Github or Bitbucket account, select the application they want to work with, then start creating the first test plan and tasks associate with the application. The user should also be able to add staging or production URL and tags for each tasks under the Settings tab. Finally, the user should be able to see their test plan progress and which browsers worked or failed.

Second, I worked on the user interface. I wanted users to have access to their test plans and tasks within the application. To achieve this, I explored the user interface of Intercom and ProductHunt.

Given the limited hours that I had per week for the project, I jumped right into creating high fidelity mockups in order to meet the project milestones. Then I played with the color palette and typography. Red, black and white was the chosen color pallete and "Source Sans Pro" was the primary font. For optimal success, I explored some of the best desktop user interface practices from Apple and Invision.

Based on the feedbacks and insights from the team, I went back to re-define the solution with new requirements.

REQUIREMENTS

  • The application won't be used by QA person only. It will be a platform for developer, QA person and the client to collaborate and test the application.
  • The user should able to create multiple release plans and copy test plans from previous release.
  • The user should able to add details about the tasks and any attachements or design assets.
{% include "includes/assets/ideate.html" %}

I drew a user flow of the application based on the new requirements to get a better understanding of how to organize the contents. For this redesign process, I skipped sketches and jumped right into creating high fidelity mockups.

First, I redesigned the user flow of the application. The user should still able to login using their Github or Bitbucket account, select the application that they want to work with, then start creating the first release version of the application. The user should then be able to start creating new test plans, tasks and detail information about their tasks. Once the user created a new release plan, they would have the option to add test plans or tasks from the previous release.

Second, I started playing around with the red "add button". I really like the concept of the floating button from the Google Material Design.The add button is now located at the same position of view port of the application. The size of the button will also help indicating whether you are adding a release plan, a test plan or a task.

Finally, I created two different views: task view and QA view. The developer will use the task view to create test plans and tasks. Once they are done with a specific task or test plan, they can mark it as "Complete" and the task will be available under QA view that indicate the task is now ready to be tested.

LESSON LEARNED

This project taught me how to carefully think about the target audience for the application. Muliple users with different goals use applications differently. I therefore designed the a simple application that anyone in our target audience could use.

I also learned that even though there are limited hours in a week, I could still meet the deadline of major milestones. Since I only worked with the small Barbershop team, I collaborated and share my ideas without wireframes. It proved, to me, that the design thinking process is very flexible and can repeat as much as possible. Design is a very agile process.
